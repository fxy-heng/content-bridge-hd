# ContentBridge 当前状态与后续计划

## 1. 系统概览

ContentBridge 是一个多平台内容发布工具，帮助创作者将同一份原始内容自动适配到公众号、知乎、B 站、小红书等平台，支持一键发布（模拟/真实）并追踪发布状态。

### 技术栈

| 层 | 当前实现 | 说明 |
|---|---|---|
| 前端 | HTML + CSS + JS ES Modules | 纯静态应用，无第三方运行依赖 |
| 核心引擎 | `src/core/` 模块 | 适配器、发布器、校验、模板、策略、日历、报告、快照 |
| 后端 | Express (Node.js) | 位于 `backend/`，负责真实发布 API |
| 测试 | Node.js 内置断言 | 单元测试 + 语法检查 + 烟测 |
| 存储 | 浏览器 LocalStorage + 后端本地 JSON | 前端存放发布日志/草稿，后端存放凭证 |
| CI | GitHub Actions | 语法检查 + 单元测试 + 烟测 |
| PWA | Service Worker + Manifest | 离线缓存核心资源，支持安装入口 |

### 平台支持矩阵

| 平台 | 适配 | 校验 | 模拟发布 | 真实发布 |
|---|---|---|---|---|
| 公众号 (WeChat) | Y | Y | Y | Y — 微信官方 API（需认证公众号） |
| 知乎 (Zhihu) | Y | Y | Y | Y — Puppeteer 浏览器自动化 |
| B 站 (Bilibili) | Y | Y | Y | Y — Puppeteer 浏览器自动化 |
| 小红书 (RedNote) | Y | Y | Y | 浏览器服务已编写，路由待完善 |
| 自定义平台 | Y | Y | Y | — |

## 2. 已完成功能

### 内容创作
- 原始内容编辑（标题、正文、标签、封面、目标受众、品牌语气、行动引导）
- 内容模板库（工具评测、教程指南、案例复盘）
- Markdown 文件导入生草稿
- 草稿版本管理（最多 12 个本地快照）

### 平台适配
- 四个内置平台适配器，各自独立风格（长文/问答/视频简介/种草笔记）
- 自定义平台注册与通用适配器
- 平台预设导入/导出
- 平台级标题备选生成
- AI 提示词包生成

### 发布工作流
- 平台限制校验（标题长度、标签数量、正文长度、封面提示）
- 发布准备度看板（可发布 / 需优化 / 阻塞）
- 发布策略建议（推荐主平台、优化顺序）
- 一键模拟/真实发布（自动回退：后端不可用或未配置凭证时退为模拟）
- 排期发布（设置未来时间自动标记为 scheduled）
- 发布日志（本地持久化，含状态、时间、失败原因）

### 导出与报告
- 各平台 Markdown 独立导出
- 工作区 JSON 导入/导出
- 日历导出 `.ics`
- CSV 报告（发布准备度 + 发布日志）
- 规则说明 Markdown 导出

### 工程质量
- 无第三方运行依赖的前端
- 单元测试覆盖适配器和发布流程
- 烟测脚本覆盖前端和后端
- GitHub Actions CI 自动检查
- 完整文档：README、架构设计、使用指南、Demo 脚本、API 契约、评审对照表

## 3. 与原始计划的差距

原 [PROJECT_PLAN.md](./PROJECT_PLAN.md) 计划使用 React + TypeScript + Vite 技术栈，当前实现改为纯静态 HTML + CSS + JS ES Modules，降低了依赖复杂度，功能完成度反而超出原计划。

原 [PR_COMMIT_PLAN.md](./PR_COMMIT_PLAN.md) 规划的 15 个 PR，实际通过更紧凑的提交节奏完成了所有核心功能，且额外交付了：
- PWA 离线支持
- 小红书浏览器自动化服务
- 后端烟测脚本
- 评审规则对照文档

## 4. 当前存在的问题

1. **小红书真实发布未完成**：`backend/services/rednote-browser.js` 和 `backend/routes/rednote.js` 已编写，但路由集成不完整，缺少完整的登录→发布流程验证。
2. **知乎真实发布选择器稳定性**：依赖知乎专栏创作中心页面结构，DOM 变动可能导致 `manual_required` 回退。
3. **B 站真实发布选择器同理**：创作中心页面更新后选择器可能失效，需持续维护。
4. **浏览器自动化无任务队列**：并发发布多个平台时没有排队机制，可能导致浏览器实例冲突。
5. **凭证存储为明文本地 JSON**：`backend/data/` 中的凭证未加密。
6. **前端为纯静态应用**：组件和状态管理较分散，随着功能增长维护成本会上升。
7. **缺少真实 AI 接入**：当前 AI 提示词包只生成 prompt 文本，未实际调用模型 API 进行内容改写。
8. **无团队协作功能**：单用户模式，无账号体系、审核流或多角色管理。

## 5. 下一步工作

### 第一阶段：补全与稳定（优先级：高）

- [ ] **完善小红书真实发布**：验证 `rednote-browser.js` 的登录→编辑→提交流程，补全 `routes/rednote.js` 路由集成。
- [ ] **加固浏览器自动化**：为 `zhihu-browser.js` 和 `bilibili-browser.js` 增加选择器降级策略，页面结构变化时给出更明确的诊断信息。
- [ ] **增加真实发布任务队列**：避免多平台同时触发浏览器实例冲突。
- [ ] **凭证加密存储**：使用环境变量密钥对 `credentials-store.js` 中的凭证做对称加密。
- [ ] **增加后端集成测试**：用 Vitest 或 Node 内置断言覆盖真实发布路由的关键路径。

### 第二阶段：AI 真实接入（优先级：中）

- [ ] **增加 AI Provider 配置**：支持用户填入 API Key、选择模型（如 Claude、GPT）。
- [ ] **实现 AI 改写调用**：将当前提示词包发送给模型服务，实际生成平台适配内容。
- [ ] **AI 与规则引擎对比模式**：用户可在规则适配版本和 AI 改写版本之间切换或对比。
- [ ] **AI 改写历史保存**：记录不同改写版本，方便回看和选择。

### 第三阶段：工程升级（优先级：中）

- [ ] **前端框架迁移评估**：评估是否需要迁移到 React/Vue + TypeScript，权衡当前纯静态方案的维护成本和迁移收益。
- [ ] **后端路由模块化重构**：将平台路由的公共逻辑（登录态检查、错误处理）抽取为中间件。
- [ ] **增加 E2E 测试**：覆盖从前端输入到后端发布的完整流程。

### 第四阶段：团队协作（优先级：低-长期）

- [ ] **用户账号系统**：支持注册/登录，区分编辑、审核、发布角色。
- [ ] **云端草稿同步**：草稿不再仅存 LocalStorage，同步到后端存储。
- [ ] **审核工作流**：内容编辑 → 审核 → 发布的 PR 式流程。
- [ ] **发布日历与批量排期**：团队视角的发布排期看板。

### 第五阶段：数据闭环（优先级：低-长期）

- [ ] **发布效果数据回收**：对接各平台内容数据（阅读量、互动量）。
- [ ] **标题/标签效果对比**：跨平台对比不同版本的表现。
- [ ] **最佳发布时间推荐**：基于历史数据训练或规则推算。
- [ ] **自动生成复盘报告**：周期性的内容表现总结与建议。

## 6. 技术债务备忘

- `src/core/` 模块较多且扁平，考虑按领域拆分子目录（如 `adapters/`、`publishing/`、`export/`）。
- `backend/server.js` 路由挂载方式直接，平台增多后考虑动态路由注册。
- 前端 `src/app.js` 为集中式入口，随着功能增长可能需要拆分为多个入口模块。
- 浏览器自动化依赖 Puppeteer 自带的 Chromium，镜像体积大，考虑支持系统 Chrome 或 playwright。
- `.github/workflows/ci.yml` 仅做语法检查，未运行后端测试（因需要浏览器环境）。
